# Data Schema & API Ingestion

This document explains the structure of the data that this standalone repository consumes, how it was originally generated by the SSCAP hardware, and how the Vercel API processes it before making it available to us.

## 1. The Raw Hardware Input
The SSCAP hardware sensors (Tlaloques) send extremely lightweight HTTP POST requests to the Vercel API. 

Depending on the sensor type, the raw JSON payload looks like one of the following:

**Endpoint: `/api/utilizado` & `/api/captado` (Flowmeters/Rain Gauges)**
```json
{
  "tlaloque_id": "848970e2-...", // Unique UUID for the sensor system
  "pulses": 12,                  // Raw count of magnetic pulses
  "used_at": "2024-03-20 16:05:00" // (Or "catched_at") The local time the event occurred
}
```

**Endpoint: `/api/nivel` (Water Level Sensors)**
```json
{
  "tlaloque_id": "848970e2-...",
  "meters": 1.25,                // Distance measurement
  "catched_at": "2024-03-20 16:10:00"
}
```

---

## 2. The Vercel API Processing
The existing Vercel Next.js application does **not** process this data into Liters or VWB tokens. Instead, its only job is to firmly stamp contextual metadata onto the reading and save it securely in Vercel Blob storage.

When the Vercel API receives the POST request, it:
1. Generates a unique UUID (`id`).
2. Generates an exact ISO 8601 server timestamp (`created_at`).
3. Extracts environmental data from the HTTP Headers (`x-real-ip`, `x-vercel-ip-city`, `x-vercel-ip-country`, `user-agent`).
4. Wraps the original payload inside a `data` object, adds the `metadata`, and saves it as a `.json` file.

---

## 3. The Downloader Output (Our Input)
When the standalone blockchain repository wants to sync, it simply calls:
`GET https://sscap-api.vercel.app/api/download`

This specific Vercel endpoint is designed to **accumulate all the historical data**. It traverses the entire Blob storage, finds all the individual `.json` files, and packages every single recorded sensor reading into a single massive JSON array response.

You can view a real-world example of this response in the `example_data.json` file in this directory. 

### Schema Breakdown of `example_data.json`

```json
{
  "total_blobs_found": 6,
  "sscap_records_processed": 6,
  "data": [
    {
      "filename": "sscap/captado/2026-03-20T11-18-58.324Z-d2df08ad...",
      "uploadedAt": "2026-03-20T11:18:59.000Z", // Vercel Blob System Timestamp
      "content": {
        "data": { // <--- THE ORIGINAL HARDWARE PAYLOAD
          "tlaloque_id": "test-preserved",
          "pulses": 8,
          "catched_at": "2024-03-20 16:05:00"
        },
        "metadata": { // <--- ADDED BY THE VERCEL SERVER
          "ip": "95.147.68.4",
          "city": "Wandsworth",
          "country": "GB",
          "userAgent": "curl/8.7.1"
        },
        "id": "d2df08ad-98f1-4e9f-83ac-7310452aa86d", // Unique record ID
        "updated_at": "2026-03-20T11:18:58.324Z",
        "created_at": "2026-03-20T11:18:58.324Z" 
      }
    }
  ]
}
```

## 4. How the Standalone Repo Uses This
1. **Tier 1 Anchoring**: The script takes the `data` array from the JSON above, extracts only the new records for today, minifies the array, and posts it to **Arweave**.
2. **Tier 2 VWB Calculation**: The quarterly oracle script looks at `content.data.pulses` and `content.data.tlaloque_id`. It maps the `tlaloque_id` to its geographic basin (or falls back to `content.metadata.city`), runs the flowmeter conversion math, and aggregates the total VWB Liters for the Base L2 smart contract.
